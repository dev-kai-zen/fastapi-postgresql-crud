# Turning this repo into your own boilerplate

This guide is for **junior developers** (and anyone new to the layout): it explains **why** we split files into folders, **what** each piece does, and **how** to add a new feature without cramming everything into one `main.py`.

**Boilerplate** here means: a **starting template** you copy for each new project. You keep the same structure (config, database, routes, features) and swap in your own business logic.

---

## When to use this guide

**Checkpoint:** you have already worked through the lesson track up to **[8. APP_CONNECT_REDIS.md](8.%20APP_CONNECT_REDIS.md)**. In practice that means:

- Postgres and Redis run (often via Docker), and your `.env` has working `DATABASE_URL` and `REDIS_URL`.
- You understand `get_db` and a simple ŌĆ£ping the databaseŌĆØ route, and `get_redis` with a ŌĆ£ping RedisŌĆØ route.

If that is not true yet, finish Guides **[1](1.%20INITIAL_SETUP.md)**ŌĆō**[8](8.%20APP_CONNECT_REDIS.md)** first; this document **reorganizes** the code into a scalable layoutŌĆöit does not replace those guides.

---

## Big picture: what we are changing

| Before (tutorial style) | After (boilerplate style) |
| ------------------------ | ------------------------- |
| Many `@app.get(...)` handlers in one `main.py` | Small `main.py` that only **creates** the app and **attaches** routers |
| Strings and secrets scattered in code | One **`config.py`** that reads **environment variables** / `.env` |
| SQL details mixed with HTTP code | **`db.py`** owns the DB connection; routes ask for a session with **`Depends(get_db)`** |
| New endpoints added at the bottom of one file | Each **feature** gets its own folder (models, routes, service, ŌĆ”) |

**Purpose:** easier testing, clearer ownership of code, and room to grow without merge conflicts in a single giant file.

---

## Step 1 ŌĆö Create the folder structure

**Purpose:** separate **infrastructure** (how the app talks to the world) from **features** (what your product actually does).

**Create these directories** (from the **repository root**):

```bash
mkdir -p app/routes
mkdir -p app/features/app_testing
mkdir -p app/features/your_feature/{models,schemas,routes,controller,service,repository}
```

**What each part is for:**

| Path | Role in plain English |
| ---- | --------------------- |
| `app/routes/` | One place to **register** all HTTP routers (so `main.py` stays tiny). |
| `app/features/app_testing/` | Health checks and ŌĆ£smoke testŌĆØ endpoints (`/health`, `/db-ping`, ŌĆ”)ŌĆönot your product logic. |
| `app/features/your_feature/` | **Template** for a real feature. Rename `your_feature` to something real (`orders`, `users`, `crud`, ŌĆ”). |
| `models/` | Database table shapes (SQLAlchemy). |
| `schemas/` | Request/response shapes (Pydantic)ŌĆöwhat JSON the API accepts and returns. |
| `routes/` | URL paths and HTTP methods; calls the controller. |
| `controller/` | Thin glue between HTTP and the service (optional but keeps routes readable). |
| `service/` | Business rules (ŌĆ£what should happenŌĆØ). |
| `repository/` | Database queries (SQLAlchemy), hidden from the route layer. |

**Note:** This repo uses **`crud`** as the example feature (`items`). When you start fresh, pick one folder name and stick to it.

Add empty packages where Python needs them:

```bash
touch app/routes/__init__.py
touch app/features/app_testing/__init__.py
touch app/features/your_feature/__init__.py
```

## Step 2 ŌĆö Centralize settings in `config.py`

**Purpose:** the app reads **one** `Settings` object. Values come from **environment variables** (and `.env` in development), not hard-coded strings. That matches how Docker Compose and production hosts inject secrets.

**Create or update** `app/config.py`:

```python
from pathlib import Path

from pydantic_settings import BaseSettings, SettingsConfigDict

_BASE_DIR = Path(__file__).resolve().parent.parent


class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=_BASE_DIR / ".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )

    app_name: str  # from env APP_NAME
    environment: str = "development"

    api_v1_prefix: str = "/api/v1"

    database_url: str  # from env DATABASE_URL
    redis_url: str  # from env REDIS_URL

    jwt_secret_key: str  # from env JWT_SECRET_KEY
    jwt_algorithm: str  # from env JWT_ALGORITHM
    access_token_expire_minutes: int  # from env ACCESS_TOKEN_EXPIRE_MINUTES
    refresh_token_expire_days: int  # from env REFRESH_TOKEN_EXPIRE_DAYS


# Env / .env supply fields; pyright cannot see that at call time.
def get_settings() -> Settings:
    return Settings()  # type: ignore[call-arg]
```

**Why `_BASE_DIR`?** `Path(__file__)` is the `config.py` file. `.parent.parent` is the **project root**, where `.env` usually livesŌĆöso the app finds `.env` no matter which folder you run tests from.

**JWT fields:** used when you add authentication later; they are **settings only** until you implement login/tokens.

**You can add more fields** any time (e.g. `cors_origins`). Copy new names into [.env.example](../.env.example) so teammates know what to set.

## Step 3 ŌĆö Database engine and `get_db` in `db.py`

**Purpose:** **one shared engine** and **one session factory** for the whole app. Routes do not call `create_engine` themselvesŌĆöthey receive a session through FastAPIŌĆÖs **`Depends(get_db)`**, which opens a session for the request and **closes** it afterward.

**Create or update** `app/db.py`:

```python
from collections.abc import Generator

from sqlalchemy import create_engine
from sqlalchemy.orm import Session, sessionmaker

from app.config import get_settings

engine = create_engine(
    get_settings().database_url,
    pool_pre_ping=True,
)

SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)


def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

**What changed vs putting SQL in `main.py`?** Database setup lives in **one module**. If the URL or pool settings change, you edit **`db.py`** only.

**`pool_pre_ping=True`:** before using a connection from the pool, SQLAlchemy checks it is still aliveŌĆöhelpful when a Docker DB restarts.

## Step 4 ŌĆö Redis client helper `redis_client.py`

**Purpose:** same idea as `get_db`ŌĆöroutes that need Redis use **`Depends(get_redis)`** instead of creating a client manually.

**Create** `app/redis_client.py` (if you do not have it yet):

```python
from collections.abc import Generator

import redis

from app.config import get_settings


def get_redis() -> Generator[redis.Redis, None, None]:
    settings = get_settings()
    client = redis.from_url(settings.redis_url, decode_responses=True)
    try:
        yield client
    finally:
        client.close()
```

Details and URL shape (`redis://:password@host:port/0`) are in **[8. APP_CONNECT_REDIS.md](8.%20APP_CONNECT_REDIS.md)**.

## Step 5 ŌĆö Move ŌĆ£app testingŌĆØ routes into a router module

**Purpose:** Guides 7ŌĆō8 often show `/db-ping` and `/redis-ping` on the **global `app`**. In a boilerplate, we move those endpoints into an **`APIRouter`**. Routers are **mini-apps** you can include under a prefix (here, `/api/v1`).

**Create** `app/features/app_testing/app_testing_routes.py`:

```python
from collections.abc import Callable
from typing import cast

import redis
from fastapi import APIRouter, Depends
from sqlalchemy import text
from sqlalchemy.orm import Session

from app.config import get_settings
from app.db import get_db
from app.redis_client import get_redis

router = APIRouter()


@router.get("/")
def read_root() -> dict[str, str]:
    return {"message": f"Connected to the API: {get_settings().app_name}"}


@router.get("/health")
async def health() -> dict[str, str]:
    settings = get_settings()
    return {"status": "Healthy", "service": settings.app_name}


@router.get("/db-ping")
def db_ping(db: Session = Depends(get_db)) -> dict[str, bool | int]:
    one = cast(int, db.execute(text("SELECT 1")).scalar_one())
    return {"ok": True, "select_1": one}


@router.get("/redis-ping")
def redis_ping(r: redis.Redis = Depends(get_redis)) -> dict[str, bool | int]:
    pong = cast(Callable[[], bool], r.ping)()
    return {"ok": True, "ping": pong}
```

**What changed?** Instead of `@app.get(...)`, we use `@router.get(...)` on a **`router`**. Later, `main.py` will **`include_router`** once for the whole API version prefix.

## Step 6 ŌĆö Register routers in `app/routes/__init__.py`

**Purpose:** one function lists **all** top-level routers for API v1. Adding a new feature = add one `include_router` lineŌĆöno need to open `main.py` for every feature.

**Create or update** `app/routes/__init__.py`:

```python
from fastapi import APIRouter

from app.features.app_testing.app_testing_routes import router as app_testing_router
from app.features.your_feature.routes.your_feature_routes import router as your_feature_router


def register_v1_routes() -> APIRouter:
    router = APIRouter()
    router.include_router(app_testing_router)
    router.include_router(your_feature_router)
    return router
```

Replace `your_feature` with your real package.

**In this repository** the imports look like this (feature folder = `crud`):

```python
from fastapi import APIRouter

from app.features.app_testing.app_testing_routes import router as routes_test_router
from app.features.crud.routes.crud_routes import router as crud_router


def register_v1_routes() -> APIRouter:
    router = APIRouter()
    router.include_router(routes_test_router)
    router.include_router(crud_router)
    return router
```

**Order matters** if two routes could match the same path; put **more specific** routers before **catch-all** ones.

## Step 7 ŌĆö Keep `main.py` small: factory + optional lifespan

**Purpose:** `main.py` should only **build** the FastAPI instance and attach **versioned** routes. Database **table creation** in development can live in a **lifespan** hook so it runs once at startup.

**Example** `app/main.py`:

```python
from contextlib import asynccontextmanager

from fastapi import FastAPI

from app.config import get_settings
from app.routes import register_v1_routes


@asynccontextmanager
async def lifespan(app: FastAPI):
    settings = get_settings()
    if settings.environment == "development":
        from app.db import engine
        from app.features.your_feature.models.your_model import Base

        Base.metadata.create_all(bind=engine)
    yield


def create_app() -> FastAPI:
    settings = get_settings()
    app = FastAPI(title=settings.app_name, lifespan=lifespan)
    app.include_router(register_v1_routes(), prefix=settings.api_v1_prefix)
    return app


app = create_app()
```

**What this does:**

- **`create_app()`** ŌĆö useful for tests (you can build a fresh app) and keeps startup logic in one place.
- **`lifespan`** ŌĆö here, **`create_all`** runs only when `environment` is `development`. For production, prefer **Alembic migrations** instead of auto-creating tables.
- **`prefix=settings.api_v1_prefix`** ŌĆö all routes from Step 6 appear under **`/api/v1`** (e.g. `/api/v1/health`).

Import **`Base`** from the module where your SQLAlchemy models are defined (this repo uses `app.features.crud.models.items_model`).

## Step 8 ŌĆö Add a new feature (replace `your_feature`)

**Purpose:** each product area gets the same **six-file pattern** so everyone knows where to look.

Under `app/features/your_feature/`, create files such as:

| File (example names) | Responsibility |
| -------------------- | -------------- |
| `models/your_model.py` | SQLAlchemy models (`class YourTable(Base): ...`) |
| `schemas/your_schema.py` | Pydantic models for JSON in/out |
| `routes/your_feature_routes.py` | `APIRouter`, `Depends(get_db)`, HTTP status codes |
| `controller/your_feature_controller.py` | Map HTTP Ōåö service calls (keep thin) |
| `service/your_feature_service.py` | Business logic |
| `repository/your_feature_repository.py` | Queries and commits |

**Junior-friendly tip:** you *can* start with one file per feature while learning. As soon as a file feels crowded, **split** along this layoutŌĆöit is easier to grow into than to untangle a 500-line module later.

**In this repo,** `your_feature` is implemented as **`crud`** with `items`ŌĆöuse it as a **copy-paste template** for your next resource.
